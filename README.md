## מבנה הפרויקט והחלטות ארכיטקטוניות
* **חישוב מחיר**: מבוצע אך ורק בצד השרת בנתיב `POST /api/orders` לפי מחירי האמת ששמורים בזיכרון. צד הלקוח מציג מחיר משוער בלבד כדי למנוע הזרקת מחירים מזויפים מהדפדפן.
* **חוקיות סטטוסים**: השרת אוכף מעברי סטטוס חוקיים (`new` -> `preparing` -> `ready` -> `delivered`) ומחזיר `409` במקרה של מעבר לא חוקי.

###  כלל אישי (ספרה אחרונה 8)
* **הכלל**: אי אפשר לבחור את אותה תוספת פעמיים לאותה פיצה.
* **מימוש**: השרת משתמש ב-`Set` כדי לוודא שאורך מערך התוספות הייחודיות שווה לאורך מערך התוספות שנשלח.

##  תשובות לשאלות החובה

1. **מה ההבדל בין צד הלקוח לצד השרת במערכת שלכם?** צד הלקוח (React) מנהל את התצוגה, הקלט והאינטראקציה של המשתמש. צד השרת (Express) שומר על הלוגיקה העסקית, אימות הקלטים, חישוב המחירים וניהול המצב הזמני בזיכרון.

2. **איפה מחושב המחיר הכולל ולמה?** בצד השרת, כדי למנוע מניפולציות של משתמשים על מחירים דרך ה-Developer Tools של הדפדפן.

3. **מה קורה כאשר לקוח שולח הזמנה לא תקינה?** השרת מחזיר שגיאת `400 Bad Request` עם פירוט הבעיה ב-JSON, וצד הלקוח מציג הודעת שגיאה מסודרת למשתמש.

4. **מה קורה לאחר שהתשלום המדומה מצליח?** נוצרת הזמנה חדשה בסטטוס `new` ונוצר עבורה מספר הזמנה ייחודי, העגלה מתנקה, ומוצג פאנל ירוק עם אישור ההזמנה והמחיר הסופי מהשרת.

5. **מהו הכלל האישי שחל עליכם?** ספרה אחרונה 8: מניעת כפל תוספות זהות על אותה הפיצה.

6. **מה היה החלק הכי מאתגר בתרגיל?** הבטחת הסינכרון ומניעת מעברי סטטוס לא חוקיים ב-API (לדוגמה מעבר מסטטוס "חדש" ישירות ל"נמסר" מבלי לעבור בייצור ובשילוח).

7. **איזו החלטת תכנון אחת קיבלתם ולמה?** לשמור את ה-Menu כמקור אמת יחיד בשרת ולחשוף אותו ב-`GET /api/menu`, כך שאם המחירים ישתנו בעתיד, נצטרך לעדכן רק מקום אחד בקוד השרת.
